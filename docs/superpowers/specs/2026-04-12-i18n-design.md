# Nudge i18n 設計文件

**日期**：2026-04-12
**狀態**：設計完成，待使用者確認後進入實作規劃

## 目標

讓 Nudge（Web + Mobile）支援繁體中文、英文、日文三語言，包含 App 核心 UI、半系統文案（task status / tag 色名）、日期數字格式、以及未來的推播通知。過程採漸進式遷移、不中斷現有開發。

## 範圍決策

| 項目 | 決定 |
|---|---|
| 支援語言 | zh-TW（預設）/ en / ja |
| Source language | 中文當 source；LLM 翻譯 en/ja，人工 review |
| 涵蓋範圍 | App 核心 UI + semantic constants（TASK_STATUSES、TAG_COLORS）+ 日期/數字格式 + 推播（預留 helper）|
| **不涵蓋** | Landing page、API 錯誤訊息（改用 error code）、email |
| Web framework | `next-intl` |
| Mobile framework | 官方 `flutter_localizations` + `gen-l10n`（ARB 格式）|
| URL 結構（Web）| URL prefix：`/en/*`、`/ja/*`、`/*`（zh-TW default）|
| Locale 決定 | 使用者設定 > 系統/瀏覽器語言 > zh-TW fallback |
| Locale 持久化 | DB `users.locale` + Web cookie + Mobile SharedPreferences |
| 翻譯檔同步 | 單一 canonical JSON + 轉檔 script |
| 翻譯產生 | LLM 一次看整份 canonical → 產 en/ja，增量模式 |
| 遷移策略 | 漸進式：基礎建設 → Web/Mobile 接線 → feature by feature（settings → tasks → cards → notes）|

## 整體架構

```
             ┌────────────────────────────────────┐
             │   i18n/canonical/  (source of truth) │
             │   zh-TW.json   (你手改)             │
             │   en.json      (LLM 生成 + review)  │
             │   ja.json      (LLM 生成 + review)  │
             └─────────────────┬──────────────────┘
                               │
               ┌───────────────┴───────────────┐
               │  i18n/scripts/sync.mjs        │
               │  · Diff + LLM incremental TL  │
               │  · JSON → next-intl messages  │
               │  · JSON → Flutter ARB (flat)  │
               │  · Validate key parity + ICU  │
               └───┬───────────────────────┬───┘
                   │                       │
         ┌─────────▼──────────┐  ┌─────────▼──────────┐
         │  Web (Next.js 16)  │  │  Mobile (Flutter)  │
         │  src/messages/*.json│ │  lib/l10n/*.arb    │
         │  next-intl          │ │  gen-l10n          │
         └────────────────────┘  └────────────────────┘
```

**核心原則**：
- `i18n/canonical/` 是唯一手改的地方（zh-TW.json）
- LLM 腳本吃 zh-TW 增量產出 en/ja（只翻新增或變動的 key）
- 轉檔腳本產出 Web 和 Mobile 各自需要的格式
- CI 跑 `sync --dry` 檢查沒漂移，有漂移就 fail

## Locale 決定順序

1. 使用者設定頁選的語言（存 DB `users.locale`）
2. 若未設定：
   - Web：`accept-language` header
   - Mobile：`PlatformDispatcher.instance.locale`
3. 若不在支援清單：fallback `zh-TW`

**URL prefix（Web）**：永遠有 prefix，`/en/cards`、`/ja/cards`、`/zh-TW/cards`。root `/` redirect 到使用者偏好。

## Canonical JSON 格式

巢狀 namespace + ICU MessageFormat（next-intl 和 ARB 子集共通）：

```json
{
  "settings": {
    "title": "設定",
    "logout": {
      "button": "登出",
      "confirm": "確定要登出嗎？"
    }
  },
  "cards": {
    "count": "{count, plural, one{# 張卡片} other{# 張卡片}}",
    "deleted": "已清除 {count} 張空白卡片"
  },
  "tasks": {
    "status": {
      "inbox": "暫記",
      "backlog": "待排入",
      "in_progress": "自己處理中",
      "waiting": "等待他人",
      "done": "完成",
      "archived": "已封存"
    }
  }
}
```

## Key 命名慣例（雙形式）

由於 Flutter 官方 gen-l10n 不支援巢狀、key 必須是 Dart identifier，canonical JSON 是巢狀，轉 ARB 時 script 自動 flatten 成 camelCase：

| Canonical | Web 呼叫 | Mobile 呼叫 |
|---|---|---|
| `settings.logout.button` | `t('settings.logout.button')` | `l.settingsLogoutButton` |
| `cards.deleted` | `t('cards.deleted')` | `l.cardsDeleted` |

Script 要檢查 flatten 後的碰撞（例如 `cards.count` 和 `cardsCount` 會衝突）、fail fast。

## Sync Script 規格

**目錄**：
```
i18n/
  canonical/
    zh-TW.json          手改
    en.json             生成
    ja.json             生成
  scripts/
    sync.mjs            主腳本
    translate.mjs       LLM 翻譯模組
    transpile.mjs       JSON→ARB 模組
  .i18n-cache.json      翻譯 hash 記錄
```

**`sync.mjs` 執行順序**：
1. **Diff canonical** — 讀 `zh-TW.json` 每個 key 的 SHA256，對比 `.i18n-cache.json`；分出「新增/變動/刪除」
2. **LLM translate 增量** — 對新增/變動的 key，呼叫 Claude API 把 zh-TW 翻成 en 和 ja；保留 ICU placeholder；合併回 `en.json` / `ja.json`
3. **Transpile to web** — 複製 canonical JSON 到 `src/messages/*.json`（next-intl 直接讀）
4. **Transpile to mobile** — 把 canonical JSON flatten 成 ARB 格式（camelCase key + `@key` metadata）、輸出到 `mobile/lib/l10n/*.arb`
5. **Validate** — key 集合一致、ICU 語法正確、無未翻譯的中文滲進 en/ja
6. **Update `.i18n-cache.json`**

**CLI**：
```bash
pnpm run i18n:sync              # 全部
pnpm run i18n:sync --dry        # 只 diff，CI 用
pnpm run i18n:sync --skip-llm   # 不翻譯只轉檔
pnpm run i18n:sync --retranslate <key>  # 強制重翻某 key
```

**LLM 翻譯策略**：
- 一次送整份 canonical JSON 給 Claude，讓 LLM 看到完整 context
- Prompt 明確要求：保留 ICU placeholder、保持語氣一致（短、動詞優先、非官方）、UI 文案風格
- 只回「新增/變動 key」的翻譯結果（由 Claude 根據 diff 清單處理）
- 支援 `@comment` 給特定 key 提示，例如 `settings.logout.button` 下加 `@button: true` 讓 LLM 知道是按鈕文字
- 重要 key 可在 `.i18n-cache.json` 標 `"locked": true`，sync 時不再自動覆蓋

## Web 端細節

**套件**：`next-intl`

**檔案結構**：
```
src/
  i18n/
    routing.ts          defineRouting({ locales: ['zh-TW', 'en', 'ja'], ... })
    request.ts          getRequestConfig，讀 user.locale / cookie
  middleware.ts         next-intl middleware + 現有 auth middleware chain
  app/
    [locale]/
      (app)/...         現有 app routes 全部搬進來
      layout.tsx
    page.tsx            root landing（不動）
messages/
  zh-TW.json
  en.json
  ja.json
```

**現有 `src/app/(app)/` 要整批搬到 `src/app/[locale]/(app)/`**。Phase 1 以單一 PR 處理，純移動。

**Locale 決定（middleware + 自訂邏輯）**：
1. URL prefix 最優先
2. 沒 prefix 且已登入 → 讀 `users.locale`，有就 redirect 到 `/${locale}/...`
3. 沒登入 → 讀 `accept-language`，在支援清單就用、否則 zh-TW
4. 設定頁切換 → `PATCH /api/me { locale }` + `router.push('/${newLocale}${currentPath}')`

**DB schema**：
```sql
ALTER TABLE users ADD COLUMN locale TEXT; -- nullable
```
Drizzle migration。

**使用 API**：
```tsx
// Server component
import { getTranslations } from 'next-intl/server';
const t = await getTranslations('settings');
return <h1>{t('title')}</h1>;

// Client component
import { useTranslations } from 'next-intl';
const t = useTranslations('settings');
```

**日期/相對時間**：用 next-intl 的 `useFormatter()`，取代現有 `date-fns/locale/zhTW` 硬編碼。

**Landing page**：暫不納入 `[locale]`，維持 root `/` 純 zh-TW。

## Mobile 端細節

**套件**：`flutter_localizations` + `gen-l10n`（官方）

**檔案結構**：
```
mobile/
  l10n.yaml                    gen-l10n 設定
  lib/
    l10n/
      app_zh.arb               生成
      app_en.arb
      app_ja.arb
      app_localizations.dart   自動生成（commit）
    core/
      locale_provider.dart     Riverpod notifier
```

**`l10n.yaml`**：
```yaml
arb-dir: lib/l10n
template-arb-file: app_zh.arb
output-localization-file: app_localizations.dart
output-class: AppL10n
```

**Locale 決定（`localeProvider`）**：
1. 讀 `SharedPreferences['nudge:locale']`
2. 沒有就讀 `PlatformDispatcher.instance.locale`
3. 若不在支援清單 → zh-TW
4. 登入後 `/api/me` 回傳 `user.locale` 與本地不同 → 跨裝置同步（以 server 為準或彈詢問？實作時定，MVP 以 server 為準覆蓋本地）

**MaterialApp 接線**：
```dart
MaterialApp.router(
  locale: ref.watch(localeProvider),
  supportedLocales: const [
    Locale('zh', 'TW'),
    Locale('en'),
    Locale('ja'),
  ],
  localizationsDelegates: [
    AppL10n.delegate,
    GlobalMaterialLocalizations.delegate,
    GlobalWidgetsLocalizations.delegate,
    GlobalCupertinoLocalizations.delegate,
  ],
  ...
)
```

**使用 API**：
```dart
final l = AppL10n.of(context);
Text(l.settingsTitle);
Text(l.cardsCount(count));  // ICU plural
```

**切換語言**（設定頁）：
- 放在「主題」區塊旁邊，類似 pattern
- 變更時 `localeProvider.setLocale(newLocale)` → 寫 SharedPreferences + `PATCH /api/me`
- Riverpod 觸發 app rebuild，文字即時切

**日期/數字**：`intl` 套件的 `DateFormat` 要依當前 locale 動態，而非硬編碼 `'zh_TW'`。需要 helper mapping（`zh-TW` → `zh_TW`、`en` → `en_US`、`ja` → `ja_JP`）。

## DB / API 變動

**Schema**：
```sql
ALTER TABLE users ADD COLUMN locale TEXT; -- null 表示未設定
```

**API**：
- `GET /api/me` 回傳加 `locale` 欄位
- `PATCH /api/me`（或 `PATCH /api/me/locale`）接受 `{ locale: 'en' }`，驗證在支援清單內、寫 DB

## 推播 i18n（Phase 9 預留）

目前沒推播系統，本階段只建立 helper 不發推播。

**策略**：預翻譯（server 發送時套用使用者 locale，payload 是成品字串）。理由：
- 實作簡單，client 收到即顯示
- 缺點：使用者切語言後，排程中的舊推播仍是舊語言 — 可接受

**Server-side t helper**（Node.js）：
```ts
// src/lib/i18n-server.ts
import zhTW from '@/messages/zh-TW.json';
import en from '@/messages/en.json';
import ja from '@/messages/ja.json';

const messages = { 'zh-TW': zhTW, en, ja };

export function t(locale: string, key: string, args?: Record<string, unknown>): string {
  // 走 canonical dot path、ICU format
}
```

推播實作時讀 `users.locale` → `t(locale, 'notification.xxx', args)` → 送 FCM/APNs。

## 風險

1. **`[locale]` 目錄搬動影響所有 Web route** — import、redirect、絕對路徑都可能要改。控制：Phase 1 獨立 PR、純搬動無邏輯、每個 route 人工跑過
2. **ARB flatten key 碰撞** — `cards.count` vs `cardsCount` 同名。sync.mjs 偵測碰撞立即 fail
3. **ICU 兩邊子集差異** — 95% 相容，少數 `select`、`selectordinal`、gender 可能要各自處理。策略：用子集優先（plural + 參數）
4. **LLM 翻譯品質** — 自創術語（「暫記」、「自己處理中」）可能翻得太官方。策略：@comment 給 hint、重要 key lock 後人工 override
5. **next-intl middleware 和現有 auth middleware chain** — 要驗證不打架。next-intl 官方有 recipe。Phase 1 實作時測
6. **Flutter locale 代碼差異** — ARB 用 `zh`，DateFormat 要 `zh_TW`。mapping helper 處理
7. **推播 helper 無實戰驗證** — 目前沒推播系統，Phase 9 理論正確但沒跑過。風險接受
8. **遷移期間新開發** — Phase 4-7 期間新 feature 應 i18n-first，避免新技術債

## Testing 策略

- **Script 單元測試**：diff 邏輯、flatten 邏輯、ICU 解析（vitest 或 node test）
- **CI**：`i18n:check --dry` 檢查 canonical ↔ generated 無漂移
- **Framework 接線**：手動三語言 golden path
- **Feature migration**：每個 PR 附三語言截圖、自己切過驗證
- **文字爆版**：人工 review，英文比中文長 1.3-1.5 倍，按鈕/toast/badge 是高風險區
- **Lint 檢查**：Phase 10 作為終驗
- **回歸**：每完成 phase，跑過先前所有 phase 的切語言路徑
